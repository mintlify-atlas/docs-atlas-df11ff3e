DevBox gives you a full development environment running in the cloud — no local installs, no dependency conflicts, no "works on my machine." You pick a runtime, Sealos provisions a container with the right tools pre-installed, and you connect from your editor of choice within seconds.

Use DevBox when you want to:

- Start a project without configuring a local environment
- Work across multiple devices without syncing state
- Share a reproducible environment with teammates
- Develop close to your production infrastructure

## Supported runtimes

DevBox supports three categories of runtime:

<CardGroup cols={3}>
  <Card title="Languages" icon="code">
    Go, Python, Node.js, Java, Rust, C
  </Card>
  <Card title="Frameworks" icon="layers">
    Gin, Hertz, Spring Boot, Flask, Next.js, Vue
  </Card>
  <Card title="Operating systems" icon="server">
    Ubuntu, CentOS
  </Card>
</CardGroup>

Each runtime comes with the compiler or interpreter, common build tools, and the correct default port already configured.

## Creating a DevBox

<Steps>
  <Step title="Open DevBox">
    From the Sealos desktop, click the **DevBox** icon to open the DevBox dashboard.
  </Step>
  <Step title="Click New DevBox">
    Click **New DevBox** in the top-right corner of the dashboard.
  </Step>
  <Step title="Choose a runtime">
    Select a language, framework, or OS template. Each template pre-configures the toolchain — for example, selecting **Next.js** gives you Node.js, npm, and the Next.js CLI.
  </Step>
  <Step title="Configure resources">
    Set the CPU and memory for your environment. The default is 2 CPU cores and 4 GiB of memory, which is sufficient for most development workloads. You can change these at any time.
  </Step>
  <Step title="Configure networking (optional)">
    Add one or more ports if your app serves traffic during development. You can expose ports over HTTP, HTTPS (via gRPC), or WebSocket. Sealos assigns a public domain automatically when you enable a public endpoint.
  </Step>
  <Step title="Create">
    Click **Create**. Your environment starts within a few seconds and appears in the DevBox list with a **Running** status.
  </Step>
</Steps>

## Connecting your editor

DevBox exposes an SSH endpoint. Any editor that supports remote SSH development can connect to it directly.

<Tabs>
  <Tab title="VS Code">
    <Steps>
      <Step title="Install the extension">
        Install the [Remote - SSH](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-ssh) extension from the VS Code Marketplace.
      </Step>
      <Step title="Get the SSH config">
        In the DevBox dashboard, click **Connect** next to your environment and select **VS Code**. Sealos copies the SSH configuration snippet to your clipboard.
      </Step>
      <Step title="Add to SSH config">
        Paste the snippet into your `~/.ssh/config` file. It includes the host alias, domain, port, and private key path.
      </Step>
      <Step title="Connect">
        Open the VS Code Command Palette (`Ctrl+Shift+P` / `Cmd+Shift+P`), run **Remote-SSH: Connect to Host**, and select your DevBox. VS Code installs its server component on first connection.
      </Step>
    </Steps>
  </Tab>
  <Tab title="Cursor">
    <Steps>
      <Step title="Get the SSH config">
        In the DevBox dashboard, click **Connect** and select **Cursor**. Sealos copies the SSH configuration to your clipboard.
      </Step>
      <Step title="Add to SSH config">
        Paste the snippet into your `~/.ssh/config` file.
      </Step>
      <Step title="Connect">
        Open Cursor's Command Palette, run **Remote-SSH: Connect to Host**, and select your DevBox.
      </Step>
    </Steps>
  </Tab>
  <Tab title="Browser (Web IDE)">
    In the DevBox dashboard, click **Connect** and select **Web IDE**. A browser-based editor opens directly — no local configuration required. This is useful when you're on a device where you can't install extensions.
  </Tab>
</Tabs>

<Note>
  The SSH private key is generated by Sealos and scoped to your DevBox. You don't need to manage key rotation manually.
</Note>

## Environment variables

You can inject environment variables into your DevBox at creation time or by editing an existing environment:

1. Open your DevBox and click **Edit**.
2. Scroll to the **Environment Variables** section.
3. Add key-value pairs directly, or reference a Kubernetes secret by pointing to a secret name and key.

Variables are available immediately in new terminal sessions and to any process started inside the container.

## Persistent storage

Your DevBox container has a writable filesystem, but you can also attach persistent volumes for data that must survive a restart:

- **Config maps** — mount a file at a specific path inside the container (useful for config files you want to version alongside your DevBox definition)
- **Volumes** — attach a persistent volume claim at a given path (useful for large datasets or build caches)

Both are configured in the DevBox edit form.

## Releasing a version

When you reach a stable checkpoint in your work, you can release a snapshot of your DevBox as a versioned image. This image can then be deployed directly via App Launchpad.

<Steps>
  <Step title="Open your DevBox">
    Click the DevBox name in the dashboard to open its detail page.
  </Step>
  <Step title="Click Release">
    Click **Release** and enter a version tag and an optional description.
  </Step>
  <Step title="Wait for the build">
    Sealos builds a container image from your current environment. The release status moves from **Pending** to **Success** when the image is ready.
  </Step>
</Steps>

Released images appear under the **Versions** tab and can be deployed from there or referenced directly in App Launchpad using the image name.

## Sharing and collaboration

Every DevBox gets a unique SSH endpoint. To give a teammate access:

1. Share the SSH config snippet from the **Connect** dialog.
2. Make sure your teammate has the SSH private key — download it from the **SSH Config** section of the DevBox detail page.

<Warning>
  The SSH private key grants full shell access to your DevBox environment. Only share it with people you trust.
</Warning>

## Stopping and resuming

DevBox environments bill based on resource consumption while running. To avoid unnecessary charges when you're not actively coding:

- **Stop**: Click **Stop** in the DevBox detail page. Sealos deallocates the compute resources but preserves your filesystem. The status changes to **Stopped**.
- **Resume**: Click **Start** to bring the environment back up. Your files and installed packages are exactly as you left them.

<Tip>
  Stopping a DevBox does not delete your code or configuration. You can safely stop environments overnight and resume them the next day.
</Tip>

## Deleting a DevBox

To permanently remove a DevBox, open its detail page and click **Delete**. This removes the container, its SSH config, and any non-persistent storage. Persistent volumes are removed unless you detach them first.

<Warning>
  Deletion is irreversible. Back up any important work before deleting a DevBox.
</Warning>
