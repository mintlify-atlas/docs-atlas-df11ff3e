After installation, your deployment is configured through a combination of environment variables passed to the installer, Helm values in the `charts/` directory, and Kubernetes ConfigMaps managed by the `cloud-config` Helm release. This page documents the configuration options that are visible in the `deploy/cloud/` directory and the install scripts.

## Configuration storage

The installer stores generated configuration (Clusterfile, TLS secrets, Higress config, and more) in:

```
/root/.sealos/cloud/
```

The Sealos cloud configuration is stored in a Kubernetes ConfigMap in the `sealos-system` namespace:

```bash
kubectl get configmap sealos-config -n sealos-system -o yaml
```

This ConfigMap is created by the `cloud-config` Helm release (from `charts/config/`) and contains the authoritative runtime configuration for all controllers and frontend components.

## TLS and domain configuration

Your cloud domain and TLS certificate are the most important configuration choices for a self-hosted deployment. The installer supports three certificate modes.

<Tabs>
  <Tab title="ACME (Let's Encrypt)">
    Enable automatic certificate issuance via ACME DNS-01 challenge. This requires your domain to support CNAME records.

    Set the following in `sealos.env` before running the installer:

    ```bash
    SEALOS_V2_CLOUD_DOMAIN="cloud.example.com"
    SEALOS_V2_ENABLE_ACME="true"
    ```

    The installer registers with the ACME DNS service and prints the CNAME record you need to add for `_acme-challenge.<your-domain>`. Add the CNAME record in your DNS provider, then press Enter to continue.

    <Note>
      The default ACME DNS host is `auth.acme-dns.io`. You can override it with `--acmedns-host=<host>` if you operate your own acme-dns server.
    </Note>
  </Tab>
  <Tab title="Bring your own certificate">
    Provide paths to an existing PEM-encoded certificate and private key:

    ```bash
    SEALOS_V2_CLOUD_DOMAIN="cloud.example.com"
    SEALOS_V2_CERT_PATH="/etc/ssl/certs/cloud.example.com.crt"
    SEALOS_V2_KEY_PATH="/etc/ssl/private/cloud.example.com.key"
    ```

    The installer reads the certificate files, base64-encodes them, and writes them as a Kubernetes Secret named `wildcard-cert` in the `sealos-system` namespace. Higress references this secret for TLS termination.

    The secret is created using this structure (managed automatically by the installer):

    ```yaml
    apiVersion: apps.sealos.io/v1beta1
    kind: Config
    metadata:
      name: secret
    spec:
      path: manifests/tls-secret.yaml
      strategy: merge
      data: |
        data:
          tls.crt: <base64-encoded-cert>
          tls.key: <base64-encoded-key>
    ```
  </Tab>
  <Tab title="Self-signed (default)">
    If you do not set `SEALOS_V2_CERT_PATH` or `SEALOS_V2_ENABLE_ACME`, the installer generates a self-signed wildcard certificate for your domain and stores it in the `wildcard-cert` secret.

    Self-signed certificates cause browser warnings and are only appropriate for internal or evaluation deployments. Browsers running against a self-signed cert deployment set `tlsRejectUnauthorized=0` automatically so that internal service-to-service calls continue to work.

    <Warning>
      Do not use self-signed certificates on internet-accessible deployments. External users will see browser security warnings and some HTTP clients will refuse the connection.
    </Warning>
  </Tab>
</Tabs>

### Changing the cloud port

The default HTTPS port is `443`. To use a non-standard port:

```bash
SEALOS_V2_CLOUD_PORT="8443"
```

This configures Higress to listen on `8443` and sets the `cloudPort` field in the `sealos-config` ConfigMap, which all frontends and controllers use to construct public URLs.

## Default region configuration

Each self-hosted Sealos installation is a single region. The region has a unique identifier (`regionUID`) generated by the installer during the initial run. It is stored in the `sealos-config` ConfigMap:

```bash
kubectl get configmap sealos-config -n sealos-system \
  -o jsonpath='{.data.regionUID}'
```

The region UID is used by billing controllers to scope resource usage to a specific region. Do not change this value after your deployment is active — doing so will orphan existing billing records.

## Database configuration

### MongoDB

MongoDB stores authentication tokens, app configuration, and workspace metadata. The `cloud-database` Helm release deploys MongoDB via KubeBlocks.

Default resource allocation (from `charts/database/values.yaml`):

```yaml
mongodb:
  version: "mongodb-6.0"
  replicas: 1
  storage: 10Gi
  resources:
    limits:
      cpu: "1"
      memory: 2Gi
    requests:
      cpu: 100m
      memory: 102Mi
```

To override these defaults, pass Helm values when running the installer or re-running the `cloud-database` Helm release:

```bash
helm upgrade --install cloud-database -n sealos-system charts/database \
  --set mongodb.storage=50Gi \
  --set mongodb.resources.limits.memory=4Gi
```

<Warning>
  MongoDB 6.0 requires a CPU that supports the **AVX instruction set**. If your CPU does not support AVX (check with `grep avx /proc/cpuinfo`), the installer automatically falls back to MongoDB 4.4.
</Warning>

After MongoDB is ready, the installer generates the connection URI from the `sealos-mongodb-conn-credential` secret in the `sealos` namespace:

```bash
kubectl get secret sealos-mongodb-conn-credential -n sealos \
  -o go-template='{{range $k,$v := .data}}{{$k}}: {{$v | base64decode}}{{"\n"}}{{end}}'
```

### CockroachDB

CockroachDB stores the billing ledger in two databases: `local` (region-scoped usage data) and `global` (cross-region aggregates). The `cloud-database` Helm release deploys CockroachDB via the CockroachDB Kubernetes operator.

Default resource allocation:

```yaml
cockroachdb:
  maxOffset: "500ms"
  storage: 5Gi
  nodes: 3
  maxUnavailable: 1
  resources:
    requests:
      cpu: 100m
      memory: 0.5Gi
    limits:
      cpu: 1000m
      memory: 2Gi
```

The CockroachDB cluster deploys three nodes with TLS enabled. The installer creates the `sealos` database user with admin privileges during initialization:

```bash
cockroach sql --certs-dir=/cockroach/cockroach-certs \
  -e "CREATE USER IF NOT EXISTS sealos WITH PASSWORD '<generated>';
      GRANT admin TO sealos;
      CREATE DATABASE IF NOT EXISTS local;
      CREATE DATABASE IF NOT EXISTS global;"
```

## Object storage

The self-hosted installer uses the KubeBlocks CSI S3 driver (`kubeblocks-csi-s3`) to provide S3-compatible object storage for user workloads. A `BackupRepo` resource is created for KubeBlocks database backups using OpenEBS local PV storage:

```yaml
apiVersion: dataprotection.kubeblocks.io/v1alpha1
kind: BackupRepo
metadata:
  name: backup
spec:
  accessMethod: Mount
  config:
    storageClassName: openebs-backup
    volumeMode: Filesystem
  pvReclaimPolicy: Retain
  storageProviderRef: pvc
  volumeCapacity: 5Gi
```

To increase the backup volume capacity, edit the `BackupRepo` resource after installation:

```bash
kubectl patch backuprepo backup --type=merge \
  -p '{"spec":{"volumeCapacity":"50Gi"}}'
```

## Network configuration

The installer configures default pod and service CIDRs. Override these if they conflict with your existing network:

```bash
SEALOS_V2_POD_CIDR="100.64.0.0/10"        # default
SEALOS_V2_SERVICE_CIDR="10.96.0.0/22"     # default
SEALOS_V2_SERVICE_NODEPORT_RANGE="30000-50000"  # default
```

These values are written into the Kubernetes `Clusterfile` before the cluster is bootstrapped. They cannot be changed after the cluster is running without rebuilding the cluster.

The Cilium CNI uses a mask size of `/24` per node by default. To change this:

```bash
SEALOS_V2_CILIUM_MASKSIZE="24"
```

## Higress ingress configuration

Higress acts as the API gateway and ingress controller. The installer generates its configuration automatically and applies it via `sealos run`. The gateway is configured as a DaemonSet listening on the host network on port 443 (or your configured `cloudPort`).

The HTTPS configuration is stored in a ConfigMap in the `higress-system` namespace:

```bash
kubectl get configmap higress-https -n higress-system -o yaml
```

This ConfigMap controls automatic HTTPS and the TLS secret reference (`sealos-system/wildcard-cert`).

## Authentication

Password-based login is enabled by default. The `passwordEnabled` field in the `sealos-config` ConfigMap controls this:

```bash
kubectl get configmap sealos-config -n sealos-system \
  -o jsonpath='{.data.passwordEnabled}'
```

The installer generates a random `passwordSalt` (64-character alphanumeric string) during first install and persists it in `sealos-config`. If the salt is regenerated, all existing password hashes become invalid and users will be locked out.

JWT tokens for internal (`jwtInternal`), regional (`jwtRegional`), and global (`jwtGlobal`) authentication are also generated once at install time and stored in `sealos-config`. Do not regenerate these tokens in a running deployment.

## Payment provider configuration

Payment providers are disabled by default. The `charts/config/values.yaml` file defines the payment configuration schema. Three providers are supported: WeChat Pay, Alipay, and Stripe.

<Accordion title="WeChat Pay">
  ```yaml
  payment:
    enabled: true
    wechat:
      enabled: true
      MchID: "<your-merchant-id>"
      AppID: "<your-app-id>"
      MchAPIv3Key: "<your-api-v3-key>"
      MchCertificateSerialNumber: "<certificate-serial-number>"
  ```

  Copy your WeChat private key to `charts/config/files/WechatPrivateKey`, then apply the Helm release:

  ```bash
  helm upgrade --install cloud-config -n sealos-system charts/config \
    --set payment.enabled=true \
    --set payment.wechat.enabled=true \
    --set payment.wechat.MchID=<your-merchant-id> \
    --set payment.wechat.AppID=<your-app-id> \
    --set-file payment.wechat.WechatPrivateKey=./path/to/WechatPrivateKey
  ```
</Accordion>

<Accordion title="Alipay">
  ```yaml
  payment:
    enabled: true
    alipay:
      enabled: true
      ALIPAY_APP_ID: "<your-app-id>"
  ```

  Alipay requires four certificate/key files. Pass them with `--set-file`:

  ```bash
  helm upgrade --install cloud-config -n sealos-system charts/config \
    --set payment.enabled=true \
    --set payment.alipay.enabled=true \
    --set payment.alipay.ALIPAY_APP_ID=<your-app-id> \
    --set-file payment.alipay.ALIPAY_ROOT_CERT=./ALIPAY_ROOT_CERT \
    --set-file payment.alipay.ALIPAY_PRIVATE_KEY=./ALIPAY_PRIVATE_KEY \
    --set-file payment.alipay.ALIPAY_CERT_PUBLIC_KEY=./ALIPAY_CERT_PUBLIC_KEY \
    --set-file payment.alipay.ALIPAY_APP_CERT_PUBLIC_KEY=./ALIPAY_APP_CERT_PUBLIC_KEY
  ```
</Accordion>

<Accordion title="Stripe">
  ```yaml
  payment:
    enabled: true
    stripe:
      enabled: true
      STRIPE_API_KEY: "<your-secret-key>"
      STRIPE_SUCCESS_POSTFIX: "/payment/success"
      STRIPE_CANCEL_POSTFIX: "/payment/cancel"
      STRIPE_CURRENCY: "usd"   # cny, usd, eur, etc.
  ```

  ```bash
  helm upgrade --install cloud-config -n sealos-system charts/config \
    --set payment.enabled=true \
    --set payment.stripe.enabled=true \
    --set payment.stripe.STRIPE_API_KEY=<your-key> \
    --set payment.stripe.STRIPE_CURRENCY=usd
  ```
</Accordion>

## Resource quota defaults

Default resource quotas for user workspaces are managed by the `sealos-cloud-account-controller`. Quotas are applied per workspace namespace and are adjustable through the cost center UI or directly via `ResourceQuota` objects.

To inspect the current quota applied to a workspace:

```bash
kubectl get resourcequota -n ns-<username>
```

## Storage path configuration

You can configure the paths used by OpenEBS and containerd before installation:

```bash
SEALOS_V2_OPENEBS_STORAGE="/var/openebs"       # default
SEALOS_V2_CONTAINERD_STORAGE="/var/lib/containerd"  # default
```

These paths must exist (or be creatable) on every node. If your nodes have a dedicated data disk mounted at a different path, set these variables before running the installer.
